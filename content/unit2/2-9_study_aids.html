Study Aids: Observability

Key Takeaways
  • The Three Pillars: Effective observability in Go, as in any modern stack, relies on the combination of Logs (for specific events), Metrics (for aggregated performance data), and Traces (for request lifecycle analysis).
  • Structured Logging is Standard: With the introduction of the log/slog package in Go 1.21, structured (JSON) logging is now a built-in, idiomatic practice, crucial for making logs machine-readable and queryable.
  • Prometheus for Metrics: The de facto standard for metrics collection is Prometheus. Go applications can be easily instrumented using the prometheus/client_golang library to expose a /metrics endpoint for scraping.
  • OpenTelemetry for Tracing: OpenTelemetry is the vendor-neutral standard for distributed tracing. It allows you to trace the path of a request across multiple microservices to identify bottlenecks and debug complex interactions.

Question

What are the three pillars of observability?

Answer

Logs: Detailed, timestamped records of discrete events.
Metrics: Aggregated numerical data over time (e.g., counters, gauges).
Traces: A record of a request's journey through a distributed system.

Question

As of Go 1.21, what is the standard library package for structured logging?

Answer

The log/slog package, which provides a fast and easy way to produce structured logs (e.g., in JSON format) without third-party dependencies.

Question

What is the primary purpose of exposing a /metrics endpoint from a Go application when using Prometheus?

Answer

The /metrics endpoint provides the application's current metrics in a standardized format. The Prometheus server periodically "scrapes" (fetches data from) this endpoint to collect and store the metrics for monitoring and alerting.

Question

What problem does Distributed Tracing solve in a microservices architecture?

Answer

It tracks the entire path of a single request as it travels through multiple different services. This helps identify which service is causing a performance bottleneck or where an error originates in a complex, distributed transaction.

Question

What is OpenTelemetry?

Answer

OpenTelemetry (OTel) is a vendor-neutral, open-source observability framework. It provides a standardized set of APIs, SDKs, and tools for instrumenting applications to generate and export telemetry data (traces, metrics, and logs), preventing vendor lock-in.
