The Global Interpreter Lock (GIL)

What is the GIL, and what kind of tasks does it primarily impact?

Answer

The GIL is a mutex in CPython that allows only one thread to execute Python bytecode at a time. It primarily impacts CPU-bound tasks, while its lock is released during I/O-bound tasks, making concurrency effective for them.

threading vs. asyncio

When would you choose the threading module over asyncio?

Answer

You would choose threading when working with blocking I/O operations and libraries that do not support the asyncio ecosystem. It allows you to add concurrency without having to rewrite your code in an async style.

Asyncio's Model

How does asyncio achieve concurrency?

Answer

asyncio uses a single-threaded event loop and cooperative multitasking. Coroutines (async functions) explicitly yield control with await when waiting for I/O, allowing the event loop to run other tasks. This can be more efficient than threading for many I/O operations.

functools.lru_cache

What is the primary purpose of the @lru_cache decorator?

Answer

Its purpose is memoization, or in-process caching. It caches the return values of a function, so if the function is called again with the same arguments, the result is returned instantly from memory, avoiding re-computation.

LRU Cache Limitations

What are the main limitations of using @lru_cache?

Answer

The cache is stored in the memory of a single process. It is not shared between processes or servers, and its contents are lost if the application restarts. It is best for caching results from pure functions.
