Study Aids: GitHub Actions for CI/CD

Key Takeaways
  • Automation in Your Repo: GitHub Actions allows you to build powerful, event-driven CI/CD pipelines directly in your repository using YAML workflow files located in the .github/workflows directory.
  • Hierarchical Structure: Workflows are triggered by events and contain one or more jobs. Each job runs on a runner and contains a sequence of steps, which can be shell commands or reusable actions.
  • Secrets Management is Critical: Never hardcode sensitive data. Always use GitHub's encrypted Secrets to securely store credentials like API keys and tokens for use in your workflows.
  • Leverage the Marketplace: Complex tasks like logging into a cloud provider, building a Docker image, or deploying to Kubernetes are made simple by using pre-built, community-vetted actions from the GitHub Marketplace.

Question

What are the five core components of GitHub Actions?

Answer

1. Workflow: The overall automated process.
2. Event: The trigger for the workflow (e.g., push).
3. Job: A set of steps running on a single runner.
4. Step: An individual command or action.
5. Action: A reusable unit of code.

Question

How do you ensure that a job in a GitHub Actions workflow only runs after other jobs have succeeded?

Answer

By using the needs keyword in the job definition. For example, needs: [lint, test] ensures the current job only starts after the lint and test jobs complete successfully.

Question

What is the correct way to handle a Docker Hub password in a GitHub Actions workflow?

Answer

Store it as an encrypted secret in the repository settings (e.g., DOCKERHUB_TOKEN) and reference it in the workflow using the syntax ${{ secrets.DOCKERHUB_TOKEN }}. Never hardcode it in the YAML file.

Question

What is a "composite action" in GitHub Actions?

Answer

A reusable action you create within your own repository (in .github/actions/) that bundles multiple workflow steps into a single, reusable unit. This helps to reduce code duplication across different workflows.

Question

What is the general process for deploying to Kubernetes from GitHub Actions?

Answer

1. Store the cluster's kubeconfig file as a GitHub Secret.
2. Add a job that uses an action to configure kubectl with that secret.
3. Use kubectl apply or a dedicated deployment action to apply your Kubernetes manifests, often updating the image tag dynamically.
